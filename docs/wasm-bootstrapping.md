# WASM Bootstrapping and Worker Setup (JavaScript)

This document explains how `nextpnr-viewer` boots the WebAssembly renderer from JavaScript and how the Web Worker is initialized. The instructions below refer to `.ts` files, but these are transpiled and bundled into a single JavaScript file during the build process.

## High-level flow

All WebAssembly code runs in a Web Worker to free up the main thread as much as possible. Most of the UI operations (sidebar etc.) are set up and maintained in JavaScript, while the actual rendering of the chip is done in wasm using an `OffscreenCanvas`.

There is only a little bit of Js provided by this library that actually runs in a Web Worker. The JS imports the `wasm-bindgen` glue code to initialize the wasm binary, and then sets up a simple RPC channel. This channel is used for the JS in the main thread to be able to communicate with the renderer running in the worker.

At runtime, the viewer is started by `NextPNRViewer` in `lib/src/index.ts` and delegated to `WorkerViewerAdapter` in `lib/src/worker.ts`.

1. Resolve the chip database URL from the selected chip (`lib/src/types.ts`).
2. Create an `HTMLCanvasElement` and transfer it to `OffscreenCanvas`.
3. Spawn a module worker from an in-memory script (`Blob` URL).
4. Inside the worker, dynamically import the wasm-pack JS glue and initialize WASM bytes.
5. Fetch and decode the chipdb binary, then construct the family-specific viewer.
6. Use RPC-style postMessage calls for all render/input/query APIs.

## 1) Chip selection and chipdb URL resolution

`lib/src/types.ts` defines two important constants:

- `CHIP_DBS`: maps architecture + size to a minimized chipdb binary URL under `lib/static/chipdb/...`.
- `SUPPORTED_DEVICES`: maps user-facing device names (for example `lp8k`, `um-45k`) to one of those URLs.

`getChipDbUrl(chip)` returns the URL for the selected `{ family, device }` pair. In `NextPNRViewer`, this URL is computed immediately in the constructor before worker creation.

## 2) Preparing config in the main thread

In `lib/src/index.ts`, `NextPNRViewer` does the following before touching WASM:

- Merges user config with `defaultConfig`.
- Converts CSS colors (hex strings) into `{ r, g, b }` objects expected by the renderer (`fromCssColor`).
- Converts optional `cellColors` the same way.
- Builds DOM layout (canvas + sidebar + loading overlay).
- Calls `canvas.transferControlToOffscreen()` to move rendering onto a worker-owned `OffscreenCanvas`.

After this, it calls `WorkerViewerAdapter.create(...)` with:

- `family`
- chipdb URL
- `OffscreenCanvas` (as transferable)
- canvas dimensions
- normalized colors and cell colors

## 3) Creating the worker from TypeScript

`WorkerViewerAdapter.create(...)` in `lib/src/worker.ts` builds worker startup URLs relative to `import.meta.url`:

- `../pkg/nextpnr_renderer.js` (wasm-bindgen JS module)
- `../pkg/nextpnr_renderer_bg.wasm` (WASM binary)

The URLs are rewritten by Rollup in the final bundle.

It then:

1. Finds the name of the correct viewer class depending on the chip family. This cannot be resolved in the worker, because the worker code does not have access to the data structure that stores this information (`types.ts`).
2. Calls `createWorkerSource(pkgModuleUrl, wasmUrl)` to generate worker code as a string.
3. Wraps it in a `Blob` and creates an object URL.
4. Starts `new Worker(workerScriptUrl, { type: 'module' })`.
5. Sends an initialization RPC (`__init__`) and transfers the `OffscreenCanvas`.

Only after `__init__` resolves does `create(...)` return the adapter.

## 4) Worker bootstrap internals

The worker source generated by `createWorkerSource(...)` is responsible for loading and wiring the renderer:

### 4.1 Dynamic module import cache

`getPkgModule()` memoizes `import(PKG_MODULE_URL)` in `pkgModulePromise`, so the JS glue module is loaded exactly once.

### 4.2 WASM initialization

`initWasm()`:

1. Fetches `WASM_URL`.
2. Throws a descriptive error if HTTP status is not OK.
3. Reads bytes with `arrayBuffer()`.
4. Calls `pkg.default({ module_or_path: wasmBytes })`.

This step initializes the wasm-bindgen runtime in the worker.

### 4.3 Viewer construction

`initViewer(family, chipdbUrl, canvas, width, height, colors, cellColors)`:

1. Awaits `initWasm()`.
2. Stores the transferred `OffscreenCanvas` and applies initial width/height.
3. Resolves the constructor using the provided viewer class name.
4. Fetches `chipdbUrl` and converts it to `Uint8Array`.
5. Creates the Rust-backed viewer instance with canvas + chipdb + colors.

From this point on, the worker owns rendering state.

## 5) RPC transport between main thread and worker

Communication is request/response over `postMessage` with an incrementing numeric `id`.

### Request shape

```ts
type WorkerRequest = {
  id: number;
  method: string;
  args: any[];
};
```

### Response shape

```ts
type WorkerResponse = {
  id: number;
  result?: any;
  error?: string;
};
```

`WorkerViewerAdapter` keeps a `pending` map of promises keyed by `id`:

- On `message`: resolve/reject the matching promise.
- On worker `error`: reject all pending calls.

The `_rpc(...)` helper is used by all public methods.

Special methods are reserved for lifecycle/control:

- `__init__`
- `__resize_canvas__` (needs to be done in JS)
- `__destroy__`

## 6) Method dispatch inside the worker

`self.onmessage` dispatches based on `method`:

- `__init__` → boot WASM + create viewer.
- `__resize_canvas__` → mutate offscreen canvas size.
- `__destroy__` → call `viewer.free()` (if present), clear references.
- otherwise → invoke renderer method via `callViewer(method, args)`.

Error handling is explicit:

- Any thrown value is converted with `serializeError`.
- Worker replies with `{ id, error }` so main-thread promises reject cleanly.

For `get_decals`, map entries are converted to arrays before posting (`Array.from(result.entries())`) and reconstructed in the adapter as `new Map(entries)`.

## 7) Main-thread lifecycle and failure handling

`NextPNRViewer` treats the adapter as an async dependency (`Promise<WorkerViewerAdapter>`):

- On successful creation: installs input listeners and sidebar UI, then hides loading overlay.
- On failure: hides loading overlay and marks viewer as dead.

Runtime protections in `lib/src/index.ts` include:

- `webglcontextlost` handling: marks viewer as dead immediately.
- Animation-frame cancellation in `destroy()`/dead path.
- Worker teardown through adapter `destroy()`.

`WorkerViewerAdapter.destroy()` performs best-effort cleanup:

1. Sends `__destroy__`.
2. Terminates worker.
3. Revokes object URL.
4. Rejects any pending RPC calls.

## Why this design

This bootstrapping approach keeps heavy rendering and wasm memory activity off the UI thread while preserving a simple TypeScript API. This is especially important in VSCode, which hates running heavy compute in the UI thread, even if that UI thread is a separate editor.
